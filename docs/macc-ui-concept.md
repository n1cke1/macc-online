# MACC-online: концепт анкеты, модели сущностей и MCP-поверхности
*Сжатая спецификация для тестовой разработки. Версия 0.4.*

Документ связывает **способ добавления меры** (через UI или через чат-агента по MCP) со **структурой сущностей** так, чтобы на выходе хватало данных для обсчёта MAC и потенциала, чтобы каждое число было аудируемо, и чтобы недоверенный ввод не мог обойти проверки.

---

## 0. Якорь

Всё существует ради двух величин:

```
MAC       = (аннуит. ΔCAPEX + ΔOPEX − ΔВыручка) / ΔВыбросы_год
Потенциал = ΔВыбросы_год × охват
```

Аннуитизация: `NPV = CAPEX − PV(ставка, срок, OPEX)`, `MAC = NPV / PV(ставка, срок, abatement) × 1000`.
Ставка, срок, год/сценарий, сетевой EF — из реестра, **не спрашиваются**.

## 1. Модель сущностей и два страта

Композируемые пресеты: технология *ссылается* на ресурсы и продукты, а не хранит готовые числа. Каждое значение несёт провенанс (§6).

Сущности расслоены по одному критерию — **образует ли это проверку или является её предметом**:

| Сущность | Ключевые поля | Страт / запись агента |
|---|---|---|
| **Мера** | см. §2 | подача — пишет |
| Локальный показатель (IN-L) | `value`, `unit`, провенанс | подача — пишет |
| **Ресурс** (энергоноситель/C-вещество) | `ef`, `price`, `unit` | смешанный — создаёт, но `ef`/`price` валидируются против референса |
| **Технология** | `capex_ud`, `eff`/`cop`, `срок`, OPEX-профиль, ссылки | смешанный — `capex_ud` валидируется |
| **Продукт / услуга** | `unit`, `service_unit` (якорь сравнимости) | смешанный |
| **Шаблон формулы** | `expr`(AST), слоты, провенанс метода | авторитет — read / `propose` |
| **Референс** | `тип`, `диапазон [min,max]`, `unit`, источник | авторитет — read / `propose` |
| **Ресурсный пул** | `caps_ref` (что каплет: Ресурс/Продукт/активность), `годовой_поток`, `unit`, сектор | авторитет — **только ссылка** |
| **Глобальный параметр** | ставка, год/сценарий, сетевой EF (врем. ряд) | авторитет — read |

> **Референс — единый концепт для обеих осей.** `{тип, диапазон, unit, источник}` сверяется с любым обратно-считанным значением: физический фактор (кг CO₂e/гол) для оси X **и** удельная стоимость ($/кВт) для оси Y.
> **Электроэнергия** — Ресурс с `ef` как временным рядом по году/сценарию, не скаляром.

## 2. Объект «Мера» (ядро = MCP-контракт)

Одна JSON Schema валидирует UI, API и MCP — единый источник для всех интерфейсов.

```yaml
measure:
  id, name, sector_ref, technology_ref
  schema_version                                # §8 — нотация = публичный read-контракт; заготовки собирают оффлайн
  scope: canonical | user-draft | scenario     # §7, §9
  owner_ref                                     # identity-claim из MCP-токена (§9)
  maturity_stage: raw | back_calc | computed    # §5 — посчитано ли
  review_status: ...                            # governance; на старте НЕ гейтит канон (§7)

  comparison:                # ось A/B выводится, не спрашивается
    service_unit_ref         # A → продукты базы и проекта валидируются на совпадение
    is_substitution: bool    # B (CCS, агро) → проверка совпадения не нужна

  flows:                     # шаги 3–4; пусто на raw
    baseline: [ {resource_ref, qty_per_unit} ]
    project:  [ {resource_ref, qty_per_unit} ]

  abatement:                 # ровно один блок по стадии (§5)
    raw:       { share, justification }
    back_calc: { share, activity_scalar:{qty,unit}, implied_factor(ro), reference_ref }
    computed:  { formula_ref, bindings, derived_share(ro) }   # §3

  economics:                 # одинаково на всех стадиях; implied $/ед сверяется с референсом (§7)
    capex   # строим − остаточная стоимость закрываемого
    opex    # Σ(Δпоток × цена) + %обслуживания
    revenue

  potential:                 # §7
    ceiling_dim: cut_resource | output_product | n_objects | activity
    pool_ref                 # ссылка на §1 пул; самосозданный пул ⇒ мера остаётся draft
    stacking_group

  provenance_rollup(ro)      # §6 — слабое звено по всей цепочке
```

`(ro)` = read-only, вычисляется движком.

## 3. Хранение формул

Формула — это **данные, а не код**: переход raw→computed есть подмена формулы абатемента, значит её надо хранить, версионировать и валидировать.

**Каноническое представление — дерево выражения над именованными ключами (AST)**, не ячейки:

```
было (Excel):  =C43*8760*C39*(C37-C38)*10^-3
AST:           mul(ref:in_2_4, 8760, ref:in_2_2, sub(ref:ef_coal, ref:ef_gas), 1e-3)
```

- Именованные ключи (`ref`/`OUT_`, уже в книге) снимают привязку к геометрии — формулы переживают вставку строк; перевод из ячеек **программный**.
- AST **компилируется обратно в HyperFormula** (уже в стеке) → собственный интерпретатор не нужен, а `etl.py --check` bit-for-bit становится **доказательством, что перевод ничего не потерял**.

**Шаблоны + escape-hatch.** 26 формул кластеризуются в несколько форм (`share`, `generation`, `delta_ef`, `activity`); экономика единообразна. Хранение: библиотека шаблонов + привязки слот→ключ на меру. Уникальные меры (EV: брутто−рост сети; CCS: циркулярна) несут собственное inline-`expr` в том же AST.

```yaml
formula_template:
  id: delta_ef
  output: abatement                 # abatement | capex | opex | revenue
  expr: {op: mul, args: [ {slot: capacity}, 8760, {slot: cf},
          {op: sub, args: [{slot: ef_in}, {slot: ef_out}]}, 1e-3 ]}
  slots: [ {name: capacity, accepts: input}, {name: cf, accepts: input},
           {name: ef_in, accepts: resource.ef}, {name: ef_out, accepts: resource.ef} ]
  provenance: {source_type: standard, citation: "..."}

# AST: лист = {ref:key} | {slot:name} | {const:n}; узел = {op, args}
# whitelist op: +,−,×,÷, SUM, LOOKUP(=INDEX/MATCH), PV  — ради аудируемости и компиляции в HF
```

Агенту MCP дозволена **привязка шаблонов** (выбрать шаблон, замапить слоты на ключи) — ограниченная безопасная поверхность. Произвольный inline-AST — привилегия `propose`/ревьюера.

## 4. Анатомия меры: гармошка для чтения, заполнения и MCP

Одна структура — три роли сразу: **просмотр** (любой пользователь раскрывает панели, чтобы понять механику и допущения меры), **заполнение** (автор/агент вносит данные в те же панели), **MCP-навигация** (LLM работает панель за панелью). Это §2-принцип «единый источник для интерфейсов», распространённый с данных на структуру.

Не линейный визард, а набор **раскрывающихся панелей (гармошка)**, заполняемых в любом порядке, в нескольких сессиях. Тип меры, ось A/B и стадия зрелости не спрашиваются — выводятся и показываются в «Обзоре» как read-only бейджи. Частично заполненная мера валидна и persist-ится (`maturity: raw`, публикуема как черновик).

**Две сквозные грани — не панели, а присутствуют в каждой:**
- **Источники** — провенанс-чип (§6) у каждого значения, везде где есть число.
- **Экономика** — CAPEX/OPEX вводятся *там, где затрата возникает*, а не одним шагом: стройка → «Что строим», избегаемое топливо → «База», новое/паразитное → «Проект». Панель «Экономика» их только **сводит**. Хранятся вместе (`economics`, §2), редактируются распределённо.

| Панель | Просмотр показывает | Заполняет | Экономика здесь |
|---|---|---|---|
| Обзор | сектор, тип, зрелость, scope, итог MAC и потенциал | `name`, `sector_ref` | MAC (сводно, ro) |
| Что строим | технология и её механика | `technology_ref` | **CAPEX стройки** |
| База (что замещаем) | замещаемые потоки, закрываемый объект | `flows.baseline` | **−OPEX избегаемое, остаточная стоимость** |
| Проект (что взамен) | новые потоки | `flows.project` | **+OPEX новое, паразитная нагрузка** |
| Снижение | как считается сокращение, implied vs референс | `abatement` (§5) | — |
| Экономика | NPV, разбивка, MAC | выручка | **сводит CAPEX/OPEX панелей → MAC** |
| Потенциал | потолок, пул, стекинг | `potential` (§7) | — |

**MCP-связка:** панель = секция схемы = единица обратной связи `validate`, которая возвращает завершённость *по панели* (см. §8). LLM навигирует по тем же панелям, что видит человек; описание панели/поля в схеме = текст, который раньше был «вопросом».

## 5. Ось зрелости

Одна шкала. «Постулировано» = ещё не доведено.

```
raw        → abatement = сектор × доля.                  Якоря нет.
back_calc  → implied = (сектор×доля) / activity_scalar.  Якорь подсвечен, сверяется с референсом.
computed   → abatement = formula(активность, фактор).    Формула РАЗВЁРНУТА: доля стала контролем.
```

Обратный счёт сверяет `implied` с `reference [min,max]`, ставит ✓/⚠ автоматически при каждом изменении доли. В коридоре → подтвердить источник → `computed`. Гибридная единица / вне коридора → backlog. Публикацию не блокирует.

## 6. Провенанс и дисциплина допущений

**Зачем.** Провенанс не украшение — на нём стоят: допуск в канон (§7 сверяет фактор с референсом; несорсенный референс = циркулярная проверка), отличие от чёрных ящиков (метка уверенности на каждом столбце кривой — «твёрдая мера» против «трёх догадок»), аудит-след при записи агентов и сам предмет ревью.

**MVP (в тесте):** лист несёт `источник + уверенность`; столбец кривой берёт **худшую** уверенность по входам. Это уже наполовину в книге (`source` ~60%, `review_status`).

```yaml
provenance:   # лист
  source_type: official_stat|literature|standard|expert_estimate|assumption|placeholder
  citation, url, date
  confidence: high|medium|low
```

Незаполненный источник = `placeholder` + ниже зрелость, не стоп. Провенанс, комментарии (decision-log, автор = `owner_ref`) и зрелость — три слоя на одном анкоре.

**На горизонте:** формальное распространение уверенности по цепочке формул (`confidence = min(входов)` до самого MAC). Для прототипа не обязательно — начинаем с «худший лист на столбце».

### Дисциплина допущений (guardrail записи агентов)

Чтобы LLM не выдумывали числа, а подтверждали их, значение допущения подаётся в одном из трёх режимов — с приоритетом переиспользованию:

```yaml
binding:
  mode: reuse | alt | new
  ref: <canonical_key>       # reuse / alt — на какое каноническое опирается
  value, divergence_reason   # alt: своё значение + ОБЯЗАТЕЛЬНО причина расхождения
  # new: значение локальное, в канон не входит (ниже)
```

- **reuse** (по умолчанию) — ссылка на существующее каноническое допущение. Гасит фрагментацию реестра (нет сотни чуть разных КИУМ).
- **alt** — разойтись с каноном можно, но только **явной альтернативой с пометкой рассогласования**. Расхождение становится видимым сигналом для ревью и работает как анти-гейминг: подкрутить допущение в свою пользу тихо нельзя — форк от канона подсвечивается.
- **new** — новых *канонических* допущений агент не создаёт (приоритет существующим). Нет каноничного аналога → значение остаётся локальным/`draft` и в канон не входит, пока эталон не появится. Это прежнее правило create-but-draft, применённое к допущениям.

## 7. Потенциал и допуск в канон (без ревью)

Курс: мера входит в канон автоматически по прохождении валидации; ревьюер на входе не нужен. Но «лимит сектором» как единственная проверка недостаточен — нужны **три guardrail’а на разных уровнях**:

```
validate(measure) →
  ✓/✗ пер-мера, физика:   обратный счёт фактора vs референс (ось X)
  ✓/✗ пер-мера, экономика: implied $/ед (CAPEX,OPEX) vs референс технологии (ось Y)
  ✓/✗ пул:        Σ по caps_ref ≤ потолок пула, через стекинг (order-independent)
  ✓/✗ сектор:     Σ снижений ≤ выбросы сектора (грубый backstop)
  предусловие:    pool_ref ∈ канонический набор  (иначе мера остаётся draft)
```

Почему так, а не только сектор:
- **Секторный цап ограничивает сумму, а не меру** — пер-мерное завышение (дегазация шахт: 1900 kt при физике ~184) проходит внутри бюджета. Ловит обратный счёт.
- **Двойной счёт живёт ниже сектора** — две меры на один уголь под секторным итогом ловит только **пуловый** цап (через `caps_ref`). Несущий guardrail — пуловый; секторный — backstop.
- **Стекинг во время расчёта кривой, не гейт на входе** — иначе «кто первый записал, тот съел бюджет». При переподписке пула потенциалы клипаются **по порядку MAC** (дешёвые берут пул первыми) → зависит от стоимости, не от порядка записи. Это и есть «гейминг сам себя гасит».
- **Потолок — годовой поток, не запас.** Инвариант: единица потолка = единица удельного ΔВыбросы (`caps_ref` пиннит сущность и её unit).

Потолок по размерностям: `cut_resource` (вход), `output_product` (выход → ссылка на Продукт), `n_objects` (установки × годовая мощность), `activity` (популяция/поток).

**Предусловие №1 для открытия записи: канонический засоренный набор пулов + референсов (факторы и удельные стоимости).** Создавать пулы агенту можно, но мера на самосозданном пуле остаётся `draft`. Новый тип фактора без референса → мера существует и публикуема как draft, но в канон не входит, пока эталон не появится.

**Внешне-собранная заготовка (§8) не привилегированна.** JSON, собранный любым клиентом оффлайн, проходит на `upsert` те же guardrail’ы (выше) и дисциплину допущений (§6) — не принимается на веру. По `schema_version` устаревшая заготовка принимается с миграцией или внятно отвергается, а не ломается тихо.

## 8. MCP-поверхность

**MCP — это слой тулов, а не транспорт.** Один набор тулов обслуживает три класса потребителей (§2-принцип «единый источник для интерфейсов»); тарифный гейт вендора касается только третьего:

- **Веб-UI напрямую** — панели §4 на сайте. Бесплатный пол: любой пользователь смотрит / заполняет / видит кривую без подписок. Это фундамент, MCP — сверху.
- **Свой хостед-чат** — MCP-клиент под вашим контролем, инференс платите вы. Разговорный режим для всех без чужой подписки; хватает небольшой модели (панели + `validate` сужают задачу до «NL → вызовы `upsert`»).
- **Внешние MCP-клиенты** (ChatGPT Business / Claude Pro / Cursor) — поверх, бонус для power-users; гейт вендора, не ваш.
- **Внешняя сборка → импорт** — read открыт всем, поэтому любой клиент (даже бесплатный, на одном чтении) выучивает нотацию по `schema://measure` + пресетам и собирает **валидную заготовку** оффлайн; запись — одним `upsert` через хостед-чат, где гейта нет. Обходит вендорский write-гейт, не нарушая его: тяжёлый разговор оплачивает чужой ассистент, финальный шаг — ваш дешёвый канал.

Тулы крупнозернисты (мердж частичного объекта), не россыпь сеттеров — идемпотентно и устойчиво к многопользовательской гонке.

```
Чтение (resources) — аноним допустим:
  list_measures        get_measure(id)      curve://current
  list_presets(kind)   list_references      list_pools
  schema://measure     # машиночитаемый контракт + пример заготовки; нотация = публичный API

Запись — страт подачи (требует OAuth, §9):
  measure.upsert(partial)        # мердж; меры + локальные IN-L
  measure.attach_source(key,..)  # провенанс
  measure.validate(id) → { missing, maturity, scope, mac, potential,
                           panels:{overview✓, build✓, baseline✓, project✓,
                                   reduction:⚠, economics✓, potential✓},
                           checks:{factor✓, economics✓, pool✓, sector✓} }

Создание сущностей — расщеплено, НЕ один create_object:
  resource.create / technology.create   # ef/capex_ud валидируются против референса
  pool.*       — НЕТ записи, только ссылка (закрытый канон)
  reference.*  — НЕТ записи, только propose (авторитет)

Prompts: add_measure, review_measure
```

Правило: **создавать можно то, что валидируется** (меры, локальные показатели, ресурсы/технологии с проверяемыми полями); **ссылаться, но не создавать — то, что образует валидацию** (пулы, референсы).

## 9. Идентичность и доступ

Авторизация расслаивается по классу потребителя (§8). Общее: **чтение и `curve.simulate` — анонимно для всех; запись — с identity.** Режим `individual auth` (не shared): identity-claim = **`owner_ref`** (без него нет атрибуции черновиков, rate-limit при no-review, авторства в decision-log).

| Потребитель | Путь идентичности | Что нужно |
|---|---|---|
| Веб-UI + свой хостед-чат | Supabase Auth напрямую → `owner_ref` | ничего сверх Supabase; **DCR не нужен** |
| Внешние MCP-клиенты | OAuth-флоу к вашему IdP | **OAuth 2.1 + Dynamic Client Registration / CIMD; bearer не принимается** (требование ChatGPT, заодно делает сервер кросс-клиентным) |

- Сервер видит не логин в Claude/ChatGPT, а личность из *вашего* IdP. Anthropic/OpenAI-аккаунт недоступен — отсюда Supabase Auth как единый IdP: identity в БД и в токене одна.
- `scope` приземляет ввод: `user-draft` / `scenario`; промоушн в `canonical` = прохождение валидатора (§7).
- Запись через внешние клиенты дополнительно гейтится тарифом вендора (write обычно Enterprise/Pro). **«Работает для не-Pro» обеспечивает веб-UI + хостед-чат — у них этого гейта нет и DCR не требуется.**

## 10. Контракт целостности (три класса изменений)

| Класс | Что меняет | `etl.py --check` |
|---|---|---|
| Рефакторинг ETL / формул (`C43 → in_2_4`) | ничего в числах | bit-for-bit сохранение |
| Созревание меры (шаблон share→physical) | значение — намеренно | сдвиг разрешён, версионируется |
| Правка провенанса | только метаданные | ни одно число не сдвинулось |

Русские ключи (`label_ru`/`ref`) — структурные идентификаторы слияния, правятся только синхронно с источником `.xlsx`.

## 11. Тестовый срез (критерий приёмки прототипа)

Реальные меры из портфеля, по стадиям, + сломанная для проверки guardrail’ов.

**A. Кормовые добавки [3-1] — кандидат в `computed`.** raw: 17.8 Mt × 0.10 = 1780 kt; скаляр 2550 тыс. голов → implied **698 кг/гол/год**, референс −30% = 600–840 → **✓**. Тест: продвижение raw→back_calc→computed, разворот формулы.

**B. Перевод ТЭЦ на газ [1.A.1-2] — `computed`.** flows: уголь→газ; шаблон `delta_ef`. Ось A: продукт совпадает (МВт·ч) → валидатор проходит, `is_substitution=false`. Тест: flow-based путь + привязка шаблона по MCP.

**C. Дегазация шахт [1.B-3] — `raw`, СЛОМАНА.** raw: 9.5 Mt × 0.2 = 1900 kt, но метан в блоке = ~184 kt → расхождение 10×. Тест: обратный счёт ставит ⚠, мера остаётся `draft`/публикуема, **в канон не входит**.

**Критерии:** движок (1) считает MAC из единого экономического ядра; (2) валидирует сервис-единицу для B; (3) ставит ✓ (A) / ⚠ (C) по физике **и** по экономике; (4) клипает потенциал стекингом order-independent при переподписке пула; (5) принимает запись только с `owner_ref` (OAuth), чтение — анонимно; (6) не блокирует публикацию ни одной, но в канон пускает только прошедшие валидатор.
