# План структурной доработки Excel перед публикацией

**Статус:** проект/план (файл `MACC_KZ_29052026_rev.xlsx` пока не трогаем).
**Дата:** 2026-06-13. **Автор предложения:** разбор архитектуры + перепись ячеек.
**Контекст:** см. `CLAUDE.md` и утверждённый план `~/.claude/plans/macc-twinkly-dove.md`.
Цель — повысить **проверяемость** и **интерактивность** модели до публикации «trust anchor»,
не нарушая принципов: только `IF/PV/SUM` + арифметика, ссылки между листами, статическое ядро.

---

## 1. Диагноз текущей структуры

Декомпозиция частично уже правильная:

* **L0 — глобальные допущения** централизованы на листе `MACC` (цены `E36–E38`, EF `E39–E42`,
  удельные CAPEX `E45–E57`, ставка `C2`). **75** ячеек в `Расчётах` корректно ссылаются на них
  (`=MACC!$E$45`). ✅
* **Выходы** каждого блока (секция `КЛЮЧЕВЫЕ ВЫХОДЫ → таблица MACC`) отделены и питают `E/F/G/H`. ✅
* **Кривая** (`MACC` I/J/K) — отдельный финансовый слой (`I = E − PV`, `J = −PV`, `K = I/J*1000`). ✅

### Узкое место: второй ярус допущений размазан и не классифицирован

Перепись колонки `C` листа `Расчёты` (776 строк):

| тип ячейки | кол-во |
|---|---|
| **захардкоженные локальные инпуты** (КПД 0.02, КИУМ 0.55, срок 20, замещ. мощность 15000, доли…) | **158** |
| ссылки на глобальные L0 (`MACC!`/`Выбросы!`) | 75 |
| внутриблочные ссылки | 112 |
| вычисления | 243 |

Плюс в формулах зашиты **смысловые коэффициенты-допущения**: `*0.3` (ПНР), `*0.01` (ТОиР),
`0.72`, `0.95`, `8760`, `1.163`… вперемешку с честными конвертерами единиц (`44/12`, `1000`, `10^6`).

Три структурных следствия:

1. **158 ключевых допущений невидимы и непроверяемы** — именно их захотел бы оспорить эксперт,
   но они не собраны ни в один реестр; вшиты в 26 блоков по ~27 строк.
2. **Инпут и вычисление живут в одной колонке `C`** — нет признака «редактируемое допущение» vs
   «производная». ETL (`scripts/etl.py`) вынужден угадывать эвристиками (`_PHYS_UNIT`, `_BAD_UNIT`,
   `extract_breakdown`, `build_physical_items`) → хрупко, ломается от опечатки в подписи.
3. **Уровни вычислений не размечены явно** — инженерный расчёт, сборка затрат и выходы идут
   подряд без формального ярлыка уровня.

## 2. Целевая ярусная архитектура

Пять явных уровней, каждый со своим «домом» и маркировкой:

| уровень | что | где живёт | статус |
|---|---|---|---|
| **L0** | Глобальные допущения (цены, EF, удел. CAPEX, WACC) | лист `MACC` 36–57, `C2` | есть ✅ |
| **L1** | Допущения по мероприятию (КПД, КИУМ, срок, мощность, доли, % ПНР/ТОиР) | **НОВЫЙ лист «Допущения»** | новое |
| **L2** | Инженерный расчёт (снижение выбросов, замещаемая мощность) | `Расчёты`, секция `РАСЧЁТ` | есть, разметить |
| **L3** | Сборка затрат (статьи CAPEX/OPEX) | `Расчёты`, секции `CAPEX`/`OPEX` | есть ✅ |
| **L4** | Выходы → MACC (`E/F/G/H`) | `Расчёты`, `КЛЮЧЕВЫЕ ВЫХОДЫ` | есть ✅ |
| **L5** | Математика кривой (NPV, disc CO₂, MAC) | `MACC` I/J/K | есть ✅ |

Главная идея: **сделать L1 явным и централизованным**, а L2–L4 — размеченными по уровню,
чтобы и человек, и ETL читали структуру детерминированно.

## 3. Доработки по приоритету

### № 1 — Колонка-классификатор в `Расчётах` (обязательный минимум перед публикацией)

Добавить в каждый блок столбец-тег уровня для каждой содержательной строки:

| тег | смысл |
|---|---|
| `IN-G` | глобальный инпут (ссылка на L0) |
| `IN-L` | локальный инпут мероприятия (L1) — **редактируемое допущение** |
| `BASE` | контекст/базлайн (напр. выбросы сектора из `Выбросы!`) |
| `CALC` | производная величина |
| `OUT`  | ключевой выход в таблицу MACC |

**Зачем:**
* ETL перестаёт угадывать — читает тег; уходят регэксп-эвристики из `etl.py` (это **упрощение**).
* Приложение точно знает, какие ячейки редактируемы (потенциальные слайдеры).
* Ревью видит границу «допущение vs расчёт» без чтения формул.

**Риск:** минимальный — **ни одна формула не меняется**, добавляется только метаданный столбец.
После — перегнать `golden` и зафиксировать новый `modelVersion`.

### № 2 — Реестр локальных допущений: новый лист «Допущения» (наибольшая ценность)

Вынести все **158** захардкоженных литералов в одну таблицу:

```
код_мероприятия · ключ · подпись(RU) · подпись(EN) · значение · единица · источник · min · max · шаг
```

Блоки `Расчётов` ссылаются на эти ячейки (`=Допущения!$E$12`) — тот же тип ссылки, что и на `MACC!`,
**никаких новых функций**, дисциплина `IF/PV/SUM` сохраняется.

**Зачем:**
* Все допущения второго яруса — в одном проверяемом месте (ядро «openness as architecture»).
* Приложение может показать **помероприятийные слайдеры**, а не только 4 глобальных + WACC.
* Источники/цитаты получают нормальный дом (сейчас в колонках `F`/`G` ad hoc).
* Убирает «магические числа» из формул.

**Риск:** средний. Перенос 158 ячеек + правка ссылок **обязан сохранить выходы бит-в-бит** —
после каждого шага гонять `py scripts/etl.py --check` (кросс-чек I/J/K против кэша Excel).

### № 3 — Извлечь смысловые коэффициенты из формул

Перенести в L1-реестр смысловые доли-допущения (`0.3` ПНР, `0.01` ТОиР, `0.72`, `0.95`…).
Чистые конвертеры единиц (`44/12`, `1000`, `10^6`, `8760` ч/год) оставить инлайн как «физику».

### № 4 — Унифицировать шаблон блока

Одинаковый порядок и набор строк-секций во всех 26 блоках, чтобы ETL индексировал по смещению,
а ревьюер сравнивал мероприятия построчно (diff). Опционально — «длинная»/tidy таблица-компаньон.

## 4. Влияние на код и риски

* Excel — верифицируемый источник; от него зависит **golden-тест (188/188)** против кэшированных
  `<v>`. Любой перенос инпутов **сохраняет численные выходы бит-в-бит**; контроль — `etl.py --check`.
* Перестройка меняет `modelVersion`/fingerprint — это нормально, перегнать golden и обновить
  `data/kz/*.json`.
* ETL переписывается с позиционно-эвристического на **метаданно-управляемый** (по тег-колонке и
  реестру «Допущения») — это **сокращает** `scripts/etl.py` (уходят `_PHYS_UNIT`/`_BAD_UNIT`/
  `extract_breakdown`/`build_physical_items`).
* Приложение: новый лист L1 даёт данные для помероприятийных слайдеров — отдельная UI-задача
  (после ETL), не блокирует публикацию ядра.

## 5. Рекомендованный порядок

1. **№ 1 (классификатор)** — минимум перед публикацией: детерминированный ETL, снятие хрупкости,
   формулы не тронуты.
2. **№ 3** — попутно с № 2 (извлечение коэффициентов в реестр).
3. **№ 2 (реестр «Допущения»)** — самый ценный для открытости; трудозатратнее, требует ре-верификации.
4. **№ 4 (унификация шаблона)** — финальная полировка.

После каждого шага: `py scripts/etl.py --check` → `py scripts/etl.py` → перегнать golden → коммит.
