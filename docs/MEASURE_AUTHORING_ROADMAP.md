# Roadmap: готовность движка / библиотеки / гида к авторингу мер агентом

> Цель: агент (через MCP) надёжно **создаёт и обновляет** меры, не теряя данные и не
> копируя достоверные числа. Источник проблем — `macc-issues.md` (17 issue аудита) + L2-хвост
> из `ISSUES_AUDIT_VALIDATION.md`. Здесь они перегруппированы **по корневой причине**, а не по
> симптому, в корни **R1–R9** (R2 по ходу разбора растворился; D12 и B5 уже закрыты ранее).
> Полное покрытие — таблицей в конце документа.
>
> Семантические правила валидации (L2) теперь тоже в roadmap — **R8** (раньше парковались в
> `docs/VALIDATION_RULES_BACKLOG.md`, который остаётся детальным приложением). Делаются после
> indicator-centric R3 и резолвера R4, на которые опираются.

## Столп I — Движок и запись

### R1 · Мера-как-версия: full-replace + ingest-гейт + single-source рефы
**Симптомы:** A2 (неизвестные ключи проглатываются), A3 (сироты `sources` при shallow-merge),
C8/C9 (достоверные числа копируются инлайн вместо `{ref}`).
**Корень:** нет единой точки приёма документа на запись — `update_measure` шлёт patch,
RPC `measure_publish` делает JSONB shallow-merge и снапшотит уже смерженный документ.
Плюс достоверные числа дублируются инлайн (напр. `base_emissions=135.3` вместо
`{ref: sub:1.A.1.coal_power#max_emissions}`).

**Архитектурное решение (синтез):**
- **Истина = документ меры** (self-contained, версионируется). Достоверные/общие числа НЕ
  копируются инлайн — они `{ref}` в библиотеку; приватные допущения меры (`eff_gain`,
  `project_factor`) остаются инлайн с provenance.
- **Апдейт = full-document replace** (агент: `get_measure` → правит весь документ → шлёт
  целиком). Merge исчезает как класс → A3 убит по построению.
- **Один ingest-гейт** `src/lib/measure/ingest.ts` (TS, переиспользуется stdio+Edge): ajv
  против `data/measure.schema.json` (форсит `additionalProperties:false` → A2) + дроп
  осиротевших `sources` (A3) + `formula_hash` (нормализованный AST) + ref-принуждение
  достоверных чисел (`findShouldRef` — обратная к `findDrift`: инлайн-значение, равное
  существующему индикатору, обязано стать ref).
- **Формульные правки = версии той же меры** (id/якоря/ревью сохраняются), НЕ новая мера.
  `formula_hash`/`change_kind` (structural vs parametric) на version-строке — различение без
  потери идентичности. Правка достоверного числа версионируется в `library_versions`
  (распространяется ре-бейком), приватного инпута/формулы — в `measure_versions`.
- **Note-паритет библиотеки:** `library_versions += note` + RPC `library_upsert` (снапшот с
  note вместо триггера) — чтобы правка достоверного числа фиксировала «почему».

**Что уже есть:** версионирование (`measure_versions` — append-only снапшот+автор; правки
формул уже хранятся), `library_versions` (автор есть, note нет), `makeResolver`/`findDrift`.

**Зона кода:** новый `src/lib/measure/ingest.ts`; `validate.ts` (`findShouldRef`); миграция
`0016` (RPC merge→replace; `measure_versions += formula_hash/change_kind/model_version`;
`library_versions += note` + `library_upsert`); `mcp/db.ts`, `mcp/server.ts` (описания,
выдача history); `package.json` (ajv).

**Открытые вопросы:** ajv в Deno Edge-бандле (codegen — externalize/standalone); порог
блокировки ref-принуждения (уникальный матч → блок; неоднозначный → warn); реконсиляция
снапшот-триггера vs RPC для библиотеки; передача `formula_hash`/`change_kind` в RPC без
смены сигнатуры.

**Отложено:** оптимистичная блокировка версии (`expected_version`); `measure_diff`+governance
на structural-правки published-мер; полная модель путей (A1 целиком, дерево computed) — это R2.

**Tradeoff:** агент теперь шлёт полный документ (дороже по токенам, но детерминированно);
жёсткий schema-гейт реджектит легаси-ключи на первой правке (нужен pre-deploy аудит).

### R2 · ~~Декларативная модель путей~~ → растворён (A1 → R1, E16 → trace-флаг)
**Статус: не самостоятельный корень.** Разбор показал: `compute` уже возвращает постатейную
разбивку (`capexItems`/`opexItems`/`physicalItems`/`localInputs`, суммы равны `capex`/`opex` по
построению), а R1/R3 единой `numericLeaves()` не требуют (`findShouldRef` ходит по `inputs[]`,
`formula_hash` хеширует AST, инварианты R3 — по библиотечным индикаторам).
**Куда ушли симптомы:**
- **A1** (`materials[N].cost_musd` несимметричен) — однострочник: дописать `cost_musd` в
  `taggablePaths`. Вливается в ingest-гейт **R1** (он и так трогает `taggablePaths`).
- **E16** (дерево computed) — почти покрыт существующей разбивкой; остаток = промежуточные узлы
  многошаговой формулы (`cap_mw × capex_ud` до `/1000`) → опциональный **`compute(…, {trace})`**.
  Мелочь, не корень. (Агент сам строил формулу, листья берёт через `resolve_ref` R4, итог
  сверяет по разбивке — остаётся лишь локализация кривого узла и трассировка чужой формулы человеком.)
- **`numericLeaves()`** (единый список путей) — опциональная гигиена против дрейфа списков;
  делать, только если дрейф снова укусит, не как отдельный корень.
**Зона кода:** `validate.ts` (`taggablePaths` += `cost_musd`); опц. `compute.ts` (`{trace}`).
**Tradeoff:** ноль — узкие правки, без рефактора-фундамента.

## Столп II — Библиотека как источник истины

### R3 · Indicator-centric библиотека: pool растворяется, единственный тип значения — indicator
**Симптомы:** C9 (три «угольных baseline»: `pool.baseline=130000` ×2, `sub#max_emissions=135300`,
`pool.annual_flow=88000`), C10 (потолки пулов — несорсенные магические литералы), C8
(emissions 135.3 ≠ generation 88.3 × ef — несведённые значения одной сущности).
**Корень:** `pool` — отдельный вид сущности, держащий числа **инлайн** (`annual_flow`,
`baselineEmissionsKt`), хотя это просто числа сущности. Целостность — это отношения между
indicator'ами одной сущности (`emissions ≈ generation × ef`), которых модель не выражает.
**Системное решение — indicator единственный тип значения, pool/ref перестают быть «особыми»:**
1. **pool растворяется в indicator.** `pool_ref` указывает на **индикатор выбросов сущности**
   (`sub#max_emissions` — общий объём, с которым мера работает; для угля = 135.3). Оба
   пуловых литерала исчезают: `annual_flow=88` — это лимит-производное (см. п.2), `baselineEmissionsKt=130`
   — мусор. Вид `pool` уходит из библиотеки (**9 → 8 видов**).
2. **Мера держит ДВА слота, оба на индикаторы** (подтверждено: `limit` уже биндится через
   `indicator_ref`): **`pool`** = общий объём выбросов, за который конкурируют меры
   (MAC-стэкинг, `→ sub#max_emissions`); **`limit`** = физический потолок собственного
   потенциала меры (мощность / объём газа / площадь, `→` физ. индикатор). Механизм — от слота,
   **роль-словарь не нужен** (идентичность несёт `key`). `stackPools` группирует по общему
   emissions-indicator-ref.
3. **ref — коридор, привязанный к indicator** (уже так: `indicator.reference_ref → Reference{range,source}`).
   Подтверждает паттерн «значение + его коридор правдоподобия живут на indicator». Коридор
   остаётся переиспользуемой типизированной сущностью (`ref.type="capex_ud:coal_to_gas"`).
4. **Инварианты целостности — над indicator'ами одной сущности по `key`:** subsector
   `emissions ≈ generation × ef` в допуске (→ C8 структурно); потолок обязан иметь
   provenance/derivation (→ C10).
5. **provenance — обязательная форма на indicator** (shape уже есть — делаем обязательной).
**Что даёт:** один тип значения (`indicator`: value+unit+owner+key+provenance+corridor), один
способ владения, один провенанс. C8/C9/C10 умирают по построению.
**Граница с backlog:** R3 = *внутрисущностная* целостность (indicator'ы одной сущности
когерентны). *Мерные* семантические проверки (классификация, честный бейдж) — в
`VALIDATION_RULES_BACKLOG.md`. C8 (owner-coherence) переезжает в R3 как структурный инвариант.
**Зона кода:** `src/lib/measure/schema.ts` (убрать `Pool`; `potential.pool_ref` → indicator-ref;
provenance обяз. на `Indicator`), `validate.ts` (`stackPools`/`sector`-чек читают indicator'ы;
owner-coherence инвариант — переиспользовать bridge/dimension-машинерию L3), `library.ts`/loader,
seed `data/kz/library/*.json` (16 пулов → ceiling-индикаторы; baseline-дубли → ref на
`max_emissions`), `upsert_library_entity` (валидация), миграция Supabase (drop `pools`/маппинг).
**Решено (разбор open questions):**
- **Q1 владелец:** owner = подотрасль (через `sector_ref`); фиктивный `caps_ref`/`activity`
  выбросить (`activity` как вид-владельца не нужен). Край: пулы на голом секторе (`2/3/4/5`) —
  завести subsector или вешать на сектор.
- **Q2 числа угля:** 135.3 = pool (выбросы); 88 = потенциал меры, ограниченный физ. индикатором
  (электрическая мощность) → это `limit`, не отдельное число пула; 130 = мусор, не приоритет,
  чистится в плановом ре-авторинге по новым правилам.
- **Q3 миграция:** механика, шаблон `limit.indicator_ref` готов; `pool_ref` перенимает; оба
  слота (`pool` + `limit`) сохраняем как две сущности в мере.
**Подзадачи R3:**
- **R3a · Ревизия владельцев библиотеки:** все 71 индикатор привязаны к правильному
  `owner_kind/owner_ref` (структурная чистка — сегодня `caps_ref` 16/16 висячие).
- **R3b · Корректность привязки меры:** валидатор — `pool_ref`/`limit`/owner-рефы резолвятся в
  *существующую и правильную* сущность; инструкция в гид (→ R6). Семантическая правильность
  классификации (продукт↔сектор↔пул) остаётся в backlog (A1.3).
**Tradeoff:** ощутимая реструктуризация (удаление вида `pool`, ретаргет `pool_ref`, миграция
16 пулов); ревизия владельцев всей библиотеки; риск слишком жёстких инвариантов (допуск + opt-out).

### R4 · Адресуемый read (фильтр + resolver) — read-сторона indicator-centric модели
**Симптомы:** E15, D13.
**Корень:** read-поверхность MCP — «дамп всего» + «итоги». `dbListLibrary` гоняет
`select('*')` по всем 9 таблицам (~270+ строк на вызов), `list_library` без параметров; нет
адресного резолвера `ref → значение+провенанс`. После R3 indicator — центр, и «найти индикатор
по владельцу» / «разрешить ref» становятся базовыми операциями авторинга → R4 это их read-сторона.
**Системное решение:**
1. **`list_library(filter)`** — `owner_ref` (первоклассный — «индикаторы подотрасли `1.A.1.coal_power`»),
   `kind`, `id`, `prefix`. Закрывает E15 + D13.
2. **`resolve_ref(ref) → { value, unit, owner, key, provenance, corridor }`** — обёртка над
   `makeResolver`; отдаёт не только значение, но и владельца, `key`, провенанс и коридор
   (`reference_ref`) — всё, что R3 делает первоклассными атрибутами индикатора. Аудит-цепочка
   из одного вызова.
**Зона кода:** `mcp/db.ts` (`dbListLibrary` — параметры фильтра вместо безусловного дампа),
`mcp/server.ts` (схемы тулов), переиспользовать `makeResolver` из `compute.ts` (тот же резолвер,
что у `findShouldRef`/R1 и инвариантов R3).
**Открытые вопросы:** формат фильтра (комбинируемость полей); `*_by_id` для мер
(`compute_by_id`/`validate_by_id`) — здесь или в движке (R2); `get_measure` по id уже есть.
**Tradeoff:** низкий риск, аддитивно (без миграции данных и изменения расчёта). Цена — новые
схемы тулов меняют `tools/list` → редеплой Edge+Worker + переподключение коннектора claude.ai;
**батчить** с другими schema-правками (R1 note-паритет), чтобы переподключаться один раз.

## Столп III — Правила разработки меры (гид)

### R5 · Гид как производное от истины
**Симптомы:** B6, B7, D14, E17.
**Корень:** гид — рукописный монолит (`skill.generated.ts` / `SKILL.md`), дрейфует от
схемы/движка/библиотеки. Retired `raw` живёт в схеме; пример 12000 vs живой 13200;
`displaced` в выводе не описан; payload монолитен.
**Системное решение:** генерировать «зеркальные» секции гида из истины — enums/deprecations
из схемы, числа-примеры из живого снапшота библиотеки, поля вывода (`displaced`) из типов
`ValidateResult`. Дрейф становится структурно невозможен. Раздавать секциями (lazy → E17).
Депрекацию `raw` довести до схемы (один источник enum).
**Зона кода:** генератор `skill.generated.ts`, `schema.ts` (enum `MaturityStage`),
`measure-ui-help.json`, `get_authoring_guide`/`guide://measure` в `mcp/server.ts`.
**Открытые вопросы:** судьба `raw` — удалить из схемы или пометить legacy-read-only?
гранулярность секций для lazy-загрузки.
**Tradeoff:** build-step для гида; удаление `raw` может затронуть legacy-меры.

### R6 · Синхронизация гида с новой моделью авторинга *(пока без конкретных решений)*
**Зачем:** правки R1–R3 меняют сам способ авторинга, а гид это ещё не отражает: апдейт =
full-replace (а не patch), достоверные числа только через `{ref}`, версии structural/parametric
(`formula_hash`/`change_kind`), note при правке библиотечной сущности, провенанс обязателен,
и — ключевое по R3 — **две сущности потенциала в мере**: `pool` = общий объём выбросов
(`→ sub#max_emissions`), `limit` = физический потолок собственного потенциала (мощность/объём/
площадь, `→` физ. индикатор); pool растворён в indicator; мера обязана биндиться к правильной
отрасли/ресурсу.
**Что:** пройтись по всему гиду (`skill`/`uiHelp`/`abatement-modes`/`potential`/`checks`) и
привести формулировки и примеры к новой модели.
**Статус:** содержательные тексты — позже (после стабилизации R1–R3); сейчас фиксируем как
обязательный шаг. Тесно связан с R5 — часть синхронизации возьмёт на себя генератор гида.
**Открытые вопросы:** определяются по итогам R1–R3.

### R7 · Маршрутизация issue от агента в GitHub
**Зачем:** агент, наткнувшись на проблему (нет коридора, кривое число, баг гейта, спорная
классификация), должен знать **куда отправить её на доработку**, а не молча обходить.
**Что:** в гид (или рядом — отдельный resource/tool) — инструкция «куда и как заводить issue»:
репозиторий GitHub, формат (id меры/сущности, ожидаемое/фактическое, версия модели), критерий
«заводить issue vs чинить самому».
**Открытые вопросы:** ручная инструкция в гиде vs тул (`open_issue` / ссылка на `gh`); целевой
репозиторий и лейблы; нужен ли шаблон issue.

### R8 · Семантический слой валидатора (L2)
**Зачем:** протокол ловит синтаксис и правдоподобие двух чисел, но пропускает семантику
«зелёными» (разбор kz-27). Раньше парковалось в `VALIDATION_RULES_BACKLOG.md`; теперь — в
roadmap, после R3 (indicator-модель/инварианты) и R4 (резолвер), на которые опираются проверки.
**Состав (из backlog):**
- **A1.3 когерентность классификации** — продукт↔сектор↔пул согласованы; мера в правильной
  отрасли/пуле (продолжает R3b: тот проверяет «ref резолвится в существующую сущность», A1.3 —
  «в *семантически правильную*»).
- **A1.5 обязательные коридоры** — `factor`/`economics` opt-in: нет `reference_ref` → проверки
  правдоподобия нет молча; коридор обязателен для чисел, материально влияющих на MAC.
- **A1.6 честный бейдж** — `eligibleForModel` включает неавтоматизированные правила или
  показывает «N/2 правил целостности не проверено».
- **A2 ортогональность осей / перегрузка имён** — панели переэкспрессируют чеки (`limit`/`pool`),
  «economics»/«provenance» означают по два разных.
- **B4 `pool: warn` vs eligibility** — привести доку к поведению (clip не блокирует).
- **C11 `economics` тихо `na`** → `incomplete`/⚠ при отсутствии коридора (смежно A1.5).
- *(C8 owner-coherence уже переехал в R3; B5 `limit` уже исправлен.)*
**Зависимости:** опирается на R3 (indicator-модель, `key`-инварианты) и R4 (resolver для
коридоров/значений) → делать после Столпа II.
**Зона кода:** `validate.ts` (чеки/панели), `guardrails.ts`; гид-синхронизация → R6.
**Открытые вопросы:** граница «структурный инвариант R3 vs семантический чек L2»; критерий
«материально влияет на MAC» для обязательности коридора.
